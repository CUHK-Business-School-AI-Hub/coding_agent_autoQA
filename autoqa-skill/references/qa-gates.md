# QA Gates

Use this reference when creating or auditing the QA control plane.

## Gate 0: Planning

Required evidence:

- Governance sources or explicitly labeled assumptions.
- Release scope and out-of-scope behavior.
- Changed-file inventory.
- Module entry/exit inventory.
- Applicable case classes with exemptions explained.
- Named happy, alternate, and failure flows.
- Automated command definitions.
- Human checks with severity if failed.
- Selected current best-practice sources for non-routine work.

The planning gate fails when placeholders remain or an exposed boundary is unnamed.

## Gate 1: File Smoke

For an executable file, require the narrowest meaningful check: parse, compile, typecheck, import, start, or invoke one observable behavior.

For a declarative file, name the covering mechanism: compiler, schema validator, consumer test, build, or generated-code verification.

Reject:

- A test that imports a file but never executes its behavior when behavior exists.
- A test created solely to increase counts.
- A broad full-suite run used as the only evidence for a newly changed file.
- Continued implementation while the narrow check is red.

## Gate 2: Module Black Box

Require all of these:

- Each public entry has at least one case.
- Each contractual output, error, write, or event is asserted by a case.
- Each declared applicable class has a case.
- Each case states a business rule and expected observable result.
- Mocks sit at genuine external or slow boundaries and preserve the complete contract.
- Critical new behavior has red-green evidence; critical existing behavior has fault-sensitivity evidence.

Do not accept line coverage as a substitute.

## Gate 3: Feature Integration

Require every named flow variant to have:

- Independent setup and cleanup.
- A scriptable entry point.
- Real internal module interaction.
- Assertions on the final business outcome and relevant durable side effects.
- A command mapped to fresh evidence.

A single end-to-end happy path does not cover alternate or failure flows.

## Gate 4: Human E2E

Require:

- Plain-language preconditions and steps.
- One observable expected result per step or check.
- Device/browser/role/data context where relevant.
- Evidence instructions.
- Predetermined failure severity.
- Retest instructions after a blocking fix.

Automated browser checks may prepare the path but cannot judge visual hierarchy, confusing copy, perceived responsiveness, or whether a workflow makes sense to its intended user.

## Evidence Integrity

Evidence is valid only when:

- It was generated by the current command definition.
- The command exited zero.
- It is within the configured age limit.
- It is from the relevant project root.
- The agent read failures rather than reporting only the exit code.

For regression tests, verify red-green by one of these methods:

- Run before the fix and observe the intended failure, then run after the fix.
- Revert the fix locally, observe failure, restore it, and observe pass.
- Apply a controlled mutation or fault injection, observe failure, restore, and observe pass.

Never leave a mutation or reverted fix in the working tree.

## Anti-Shortcut Checks

Stop and redesign when:

- Test names describe methods instead of business behavior.
- Most assertions verify mock calls.
- Mock setup is longer than the behavior under test without a clear boundary reason.
- A partial fake omits fields present in the real contract.
- The same command is listed as evidence but has not been rerun after changes.
- The agent claims all flows pass while the flow catalog contains `pending` or `out_of_scope` without approval.
- Human E2E is marked passed by an agent.
