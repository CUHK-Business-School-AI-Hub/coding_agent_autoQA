# QA 菜鸟百科

[English](rookie-qa-pedia.md) | [简体中文](rookie-qa-pedia_CN.md) | [繁體中文（香港）](rookie-qa-pedia_HK.md)

<!-- sync-key: purpose -->
这是给第一次做产品的负责人的按需扫盲材料。它不教你写测试代码，而是帮助你理解 agent 的声明、提出有效问题，并识别薄弱 QA。

可以先快速读一遍。以后在 AutoQA 计划或报告里遇到陌生词，再回来查对应章节。

<!-- sync-key: bug-failure-oracle -->
## 1. Bug、Failure 和 Test Oracle

**Bug** 或 **defect** 是产品或设计中存在的错误；**failure** 是这个错误被实际观察到的时刻，例如应用崩溃、金额算错，或一个用户看到了另一个用户的数据。

**Test oracle（测试判定依据）**是测试如何知道结果正确。只有状态码通常很弱；更完整的判定可能包括返回值、保存的记录、发出的事件、没有变化的余额，或没有泄露的数据。

值得问：“究竟观察到什么，才能证明这个行为是正确的？”

<!-- sync-key: test-levels -->
## 2. Smoke、Module、Integration 和 E2E

- **冒烟测试**检查一个小东西能否启动或完成一个代表性动作，用来快速发现明显损坏。
- **模块测试**把一个业务组件当作黑盒，挑战它的公开边界；有些仓库也叫 unit 或 component test。
- **集成测试**检查真实组件能否协同，例如 service 加数据库、命令加文件、API 加事件处理器。
- **端到端测试（E2E）**跟随完整的用户或系统旅程。
- **回归测试**保存已经发现的行为或 bug 修复，防止问题悄悄回来。
- **合同测试**检查接口两边是否对 API schema、事件、CLI 或文件格式达成一致。

不同层级回答不同问题。一个巨大的 E2E 不能替代细致的模块边界测试；很多模块测试也不能证明拼装后的完整旅程可用。

<!-- sync-key: black-box -->
## 3. 黑盒、入口和出口

黑盒测试不关心内部私有实现，只判断公开行为。

**入口**是请求模块做事的方式：函数、endpoint、命令、事件、job、文件或提交动作。**出口**是所有可观察的离开方式：结果、错误、数据库写入、事件、外部请求或合同规定的日志。

当 agent 说“模块已经覆盖”时，可以要求它展示入口出口清单，以及每项对应的测试。

<!-- sync-key: boundary-techniques -->
## 4. 边界、等价类、决策和状态

**边界值分析**测试限制值本身及其两侧，例如 0、1、最大值和超过最大值 1。

**等价类划分**把预期行为相同的输入分组，从每个有意义的组中取样，而不是遍历所有值。

**决策表**列出多种业务条件组合及预期结果，适合退款、权限、定价或资格规则。

**状态转换测试**检查允许和禁止的变化，例如草稿 -> 已提交、已付款 -> 已退款，或已关闭 -> 重新打开。

这些技术可以防止 agent 只挑友善例子。

<!-- sync-key: doubles -->
## 5. Mock、Stub、Fake 和真实组件

它们都属于 **test double（测试替身）**：

- **Stub** 返回预先准备的答案。
- **Mock** 通常记录交互，让测试检查谁调用了什么。
- **Fake** 实现一个简化但能运行的版本，例如内存邮件网关。

替身能让测试快速且可控，但也可能说谎。部分 fake 可能和真实 API 不一致；大量 mock 的测试可能只证明 mock 配置成功。AutoQA 倾向于使用真实本地组件，并只在真正的外部或慢速边界使用符合完整合同的替身。

值得问：“这个替身可能掩盖什么真实行为？”

<!-- sync-key: data-environment -->
## 6. 测试数据、Fixture 和环境

**测试数据**是测试需要的账号、记录、文件、日期和状态；**fixture** 是可重复创建这些条件的准备过程。

**环境**是软件运行的地方：本地开发、隔离测试、staging 或 production。测试不应使用真实客户数据、生产凭证或不受控的共享状态。

好的测试会创建或明确命名自己的数据，在需要时控制时间和随机性，并在结束后清理。

<!-- sync-key: flaky -->
## 7. Flaky Test 和错误结果

**Flaky test** 会在产品没有实质变化时忽过忽败，常见原因包括固定等待时间、共享数据、未控制的时间、网络依赖或执行顺序依赖。

**假阳性**表示存在缺陷却报告质量良好；**假阴性**表示产品正确却报告测试失败。

反复重跑 flaky test 直到变绿不算证据。必须定位原因，或明确报告限制。

<!-- sync-key: coverage -->
## 8. Coverage 不只是百分比

**代码覆盖率**衡量哪些行或分支被执行过，但不能证明 assertion 有意义。

**需求覆盖**把每项承诺行为映射到证据；**合同覆盖**映射输入、输出、错误和副作用；**业务流覆盖**映射完整旅程及变体。

即使代码覆盖率 100%，仍可能漏掉错误公式、薄弱断言、权限问题或未测试流程。AutoQA 把覆盖率数字当作线索，而不是发布批准。

<!-- sync-key: red-green-mutation -->
## 9. Red-Green 和 Mutation

在 **red-green** 中，新测试先因为预期原因失败，然后在实现后通过。先看到红色很重要，因为立刻通过的测试可能根本没有测试新东西。

**Mutation（变异）**是故意引入一个小错误：翻转权限、删除校验或移动边界。如果相关测试没有失败，测试就很薄弱。随后必须恢复改动并重跑测试。

AutoQA 会针对新、关键或可疑行为选择性使用这种证明，而不是盲目变异所有代码。

<!-- sync-key: evidence -->
## 10. 新鲜且可复现的证据

别人或另一个 agent 能重复，证据才有价值。它包括精确命令、环境、相关数据、时间、退出结果和失败输出。

“以前通过过”“应该能用”和一张无关绿色仪表板截图都不是新鲜证据。AutoQA 会为命令定义生成指纹并拒绝过期结果。

<!-- sync-key: severity -->
## 11. Severity、Priority 和 Stop-Ship

**Severity（严重程度）**表示缺陷对用户或系统影响多大；**priority（优先级）**表示团队选择何时处理。严重缺陷可能很容易修；低严重度问题也可能因为业务原因被优先安排。

**Stop-ship** 缺陷会阻止发布。在 AutoQA 中，安全泄露、数据损失、核心流程阻断、重大结果错误或严重无障碍障碍等未解决 P0/P1 会停止受影响测试和发布声明。

只有继续执行相互独立且安全时，P2/P3 才可以集中记录和提交。

<!-- sync-key: human-role -->
## 12. 人仍然负责什么

自动化擅长精确、可重复的观察；人仍需要判断：

- 页面看起来是完整有意图的，还是像坏掉了？
- 目标用户能否理解标签和下一步？
- 加载、滚动、键盘、触摸和响应式行为是否好用？
- 警告是否与风险匹配？
- 完整旅程在实现团队的思维模型之外是否仍然合理？

人工 QA 不是随便点击。AutoQA 会准备受控步骤和预期结果，让人的判断也成为可重复证据。

<!-- sync-key: questions -->
## 值得问 Agent 的问题

- 哪一项需求的证据最薄弱？
- 哪个公开入口或出口没有覆盖？
- 哪些非 happy-path 被判定为不适用，理由是什么？
- 新测试在通过前，是否因预期原因失败过？
- 这个 mock 或 fake 省略了哪些真实情况？
- 哪些业务流还没有执行？
- 证据多久以前生成，命令是否变化？
- 发布前有哪些事情必须由我亲自检查？
