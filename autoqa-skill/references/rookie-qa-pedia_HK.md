# QA 新手百科

[English](rookie-qa-pedia.md) | [简体中文](rookie-qa-pedia_CN.md) | [繁體中文（香港）](rookie-qa-pedia_HK.md)

<!-- sync-key: purpose -->
這是給第一次做產品的負責人按需要閱讀的入門材料。它不教你寫測試程式碼，而是幫你理解 agent 的聲明、提出有用問題，以及識別薄弱 QA。

可以先快速讀一遍。日後在 AutoQA 計劃或報告見到陌生詞，再回來看相應章節。

<!-- sync-key: bug-failure-oracle -->
## 1. Bug、Failure 和 Test Oracle

**Bug** 或 **defect** 是產品或設計裡的錯誤；**failure** 是這個錯誤實際出現的時刻，例如應用程式 crash、金額計錯，或一個用戶看到另一個用戶的資料。

**Test oracle（測試判定依據）**是測試如何知道結果正確。只有 status code 通常很弱；較完整的判定可能包括返回值、儲存記錄、發出的 event、沒有改變的結餘，或沒有洩漏的資料。

值得問：「究竟觀察到甚麼，才能證明這個行為正確？」

<!-- sync-key: test-levels -->
## 2. Smoke、Module、Integration 和 E2E

- **冒煙測試**檢查一個小部分能否啟動或完成一個代表性動作，用來快速發現明顯損壞。
- **模組測試**把一個業務組件當成黑盒，挑戰它的公開邊界；有些 repository 也叫 unit 或 component test。
- **整合測試**檢查真實組件能否協作，例如 service 加 database、command 加檔案、API 加 event handler。
- **端到端測試（E2E）**跟隨完整的用戶或系統流程。
- **回歸測試**保存已發現的行為或 bug fix，防止問題靜靜回來。
- **合約測試**檢查介面兩邊是否對 API schema、event、CLI 或檔案格式有一致理解。

不同層級回答不同問題。一個大型 E2E 不能取代細緻的模組邊界測試；很多模組測試也不能證明組裝後的完整流程可用。

<!-- sync-key: black-box -->
## 3. 黑盒、入口和出口

黑盒測試不理會內部私人實現，只判斷公開行為。

**入口**是要求模組做事的方法：function、endpoint、command、event、job、檔案或提交動作。**出口**是所有可觀察的離開方式：結果、錯誤、database write、event、外部 request 或合約規定的 log。

當 agent 說「模組已覆蓋」時，可以要求它展示入口出口清單，以及每項對應的測試。

<!-- sync-key: boundary-techniques -->
## 4. 邊界、等價類、決策和狀態

**邊界值分析**測試限制值本身及其兩旁，例如 0、1、最大值和超過最大值 1。

**等價類劃分**把預期行為相同的輸入分組，從每個有意義的組別取樣，而不是測試所有值。

**決策表**列出多種業務條件組合及預期結果，適合退款、權限、定價或資格規則。

**狀態轉換測試**檢查容許和禁止的變化，例如草稿 -> 已提交、已付款 -> 已退款，或已關閉 -> 重新開啟。

這些技術可防止 agent 只選容易通過的例子。

<!-- sync-key: doubles -->
## 5. Mock、Stub、Fake 和真實組件

它們都屬於 **test double（測試替身）**：

- **Stub** 返回預先準備的答案。
- **Mock** 通常記錄互動，讓測試檢查誰呼叫了甚麼。
- **Fake** 實現一個簡化但可運作的版本，例如記憶體內的電郵 gateway。

替身令測試快速和可控，但也可能說謊。部分 fake 可能與真實 API 不一致；大量 mock 的測試可能只證明 mock 設定成功。AutoQA 傾向使用真實本地組件，只在真正外部或慢速邊界使用符合完整合約的替身。

值得問：「這個替身可能隱藏了甚麼真實行為？」

<!-- sync-key: data-environment -->
## 6. 測試資料、Fixture 和環境

**測試資料**是測試所需的帳戶、記錄、檔案、日期和狀態；**fixture** 是可重複建立這些條件的準備程序。

**環境**是軟件運行的地方：本地 development、隔離 test、staging 或 production。測試不應使用真實客戶資料、production credential 或不受控的共享狀態。

好的測試會建立或清楚命名自己的資料，需要時控制時間和隨機性，並在完成後清理。

<!-- sync-key: flaky -->
## 7. Flaky Test 和錯誤結果

**Flaky test** 會在產品沒有實質改變時有時通過、有時失敗，常見原因包括固定等待時間、共享資料、未控制的時間、網絡依賴或執行次序依賴。

**假陽性**表示有缺陷卻報告質素良好；**假陰性**表示產品正確卻報告測試失敗。

重複執行 flaky test 直到變綠不算證據。必須找出原因，或清楚報告限制。

<!-- sync-key: coverage -->
## 8. Coverage 不只是一個百分比

**程式碼覆蓋率**衡量哪些行或 branch 曾執行，但不能證明 assertion 有意義。

**需求覆蓋**把每項承諾行為映射到證據；**合約覆蓋**映射輸入、輸出、錯誤和副作用；**業務流程覆蓋**映射完整流程及變體。

即使程式碼覆蓋率 100%，仍可能漏掉錯誤公式、薄弱 assertion、權限問題或未測流程。AutoQA 把覆蓋率數字當作線索，而不是發佈批准。

<!-- sync-key: red-green-mutation -->
## 9. Red-Green 和 Mutation

在 **red-green** 裡，新測試先因預期原因失敗，然後在實現後通過。先看到紅色很重要，因為立即通過的測試可能根本沒有測試新東西。

**Mutation（變異）**是故意引入一個小錯誤：反轉權限、刪除 validation 或移動邊界。如果相關測試沒有失敗，測試就很薄弱。之後必須還原改動並重新執行測試。

AutoQA 會針對新、關鍵或可疑行為選擇性使用這種證明，而不是盲目變異所有程式碼。

<!-- sync-key: evidence -->
## 10. 最新而且可重現的證據

別人或另一個 agent 能夠重複，證據才有價值。它包括準確指令、環境、相關資料、時間、退出結果和失敗輸出。

「以前通過」「應該可以」和一張無關的綠色 dashboard 截圖都不是最新證據。AutoQA 會為指令定義建立 fingerprint，並拒絕過期結果。

<!-- sync-key: severity -->
## 11. Severity、Priority 和 Stop-Ship

**Severity（嚴重程度）**表示缺陷對用戶或系統影響有多大；**priority（優先次序）**表示團隊選擇何時處理。嚴重缺陷可能很容易修；低嚴重度問題也可能因業務原因優先安排。

**Stop-ship** 缺陷會阻止發佈。在 AutoQA 裡，安全洩漏、資料損失、核心流程受阻、重大結果錯誤或嚴重無障礙障礙等未解決 P0/P1，會停止受影響測試和發佈聲明。

只有繼續執行互相獨立而且安全時，P2/P3 才可集中記錄和提交。

<!-- sync-key: human-role -->
## 12. 人仍然負責甚麼

自動化擅長準確、可重複的觀察；人仍需要判斷：

- 頁面看起來完整而有意圖，還是像壞掉？
- 目標用戶能否理解標籤和下一步？
- 載入、捲動、鍵盤、觸控和 responsive 行為是否好用？
- 警告是否與風險相稱？
- 完整流程離開實現團隊的思維模型後是否仍然合理？

人工 QA 不是隨意點擊。AutoQA 會準備受控步驟和預期結果，令人作出的判斷也成為可重複證據。

<!-- sync-key: questions -->
## 值得問 Agent 的問題

- 哪一項需求的證據最薄弱？
- 哪個公開入口或出口未覆蓋？
- 哪些非 happy-path 被判定為不適用，理由是甚麼？
- 新測試在通過之前，是否因預期原因失敗過？
- 這個 mock 或 fake 省略了哪些真實情況？
- 哪些業務流程仍未執行？
- 證據是多久之前產生，指令有沒有改變？
- 發佈前有哪些事情必須由我親自檢查？
